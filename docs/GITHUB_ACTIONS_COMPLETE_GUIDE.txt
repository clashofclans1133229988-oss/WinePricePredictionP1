📚 PART 1: WHAT IS GITHUB ACTIONS?
================================================================================

SIMPLE DEFINITION:
──────────────────
GitHub Actions is an automation tool that runs custom tasks when events happen
on your GitHub repository (like pushing code, creating pull requests, etc.)

ANALOGY: AUTOMATIC WORKFLOW SYSTEM
───────────────────────────────────

Traditional Development (Without GitHub Actions):
┌─────────────────────────────────────────────────────────┐
│ You push code to GitHub manually...                     │
│                                                         │

📚 PART 2: WHAT IS CI/CD?
================================================================================

CI/CD = Continuous Integration / Continuous Deployment

Pronounced: "See-I-See-D"

CONCEPTS EXPLAINED:

CONTINUOUS INTEGRATION (CI):
───────────────────────────

What is it?
  └─ Automatically test code changes immediately after pushing

How it works:
  1. You push code
  2. GitHub Actions runs all tests
  3. Results appear in seconds/minutes
  4. Green ✅ = all tests pass (safe to merge)
  5. Red ❌ = tests failed (fix before merging)

Benefit:
  └─ Catch bugs early, before they reach production

CONTINUOUS DEPLOYMENT (CD):
──────────────────────────

What is it?
  └─ Automatically deploy code to production after tests pass

How it works:
  1. Tests pass ✅
  2. Build Docker image
  3. Upload to registry
  4. Deploy to production
  5. App live automatically

Benefit:
  └─ New features reach users instantly without manual deployment

📚 PART 3: GITHUB ACTIONS CORE CONCEPTS
================================================================================

CONCEPT 1: WORKFLOWS
──────────────────

What is it?
  └─ A sequence of automated tasks defined in YAML file

Location:
  └─ .github/workflows/workflow-name.yml

Trigger:
  └─ Runs when specific events occur:
     ├─ Push to main branch
     ├─ Pull request created/updated
     ├─ Schedule (like cron job)
     ├─ Manual trigger
     └─ Other GitHub events

Example triggers:
  on:
    push:
      branches: [main]
    pull_request:
      branches: [main]

CONCEPT 2: JOBS
───────────────

What is it?
  └─ A set of steps that run on the same machine

Properties:
  ├─ Sequential (steps run one after another)
  ├─ Run on a runner (machine)
  ├─ Can have dependencies (wait for other jobs)
  └─ Can run in parallel (multiple jobs at same time)

Example:
  jobs:
    test:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v2
        - run: pip install -r requirements.txt
        - run: pytest

CONCEPT 3: STEPS
────────────────

What is it?
  └─ Individual commands or actions in a job

Two types:
  ├─ uses: (use pre-built action)
  └─ run: (execute shell command)

Example:
  steps:
    - uses: actions/checkout@v2           # Check out code
    - run: pip install -r requirements.txt # Run pip install
    - run: pytest tests/                   # Run tests

CONCEPT 4: ACTIONS
──────────────────

What is it?
  └─ Reusable code blocks (like functions)

Official actions by GitHub:
  ├─ actions/checkout@v2 (get your code)
  ├─ actions/setup-python@v2 (install Python)
  ├─ actions/upload-artifact@v2 (save files)
  └─ actions/download-artifact@v2 (get saved files)

Community actions:
  └─ Built by community, available on GitHub Marketplace

Format:
  owner/repo@version
  Example: actions/checkout@v2

CONCEPT 5: RUNNERS
──────────────────

What is it?
  └─ Virtual machines that execute your workflow

Choices:
  ├─ ubuntu-latest (Linux)
  ├─ windows-latest (Windows)
  ├─ macos-latest (Mac)
  └─ self-hosted (your own server)

Default:
  └─ GitHub provides free runners

CONCEPT 6: EVENT TRIGGERS
──────────────────────────

Common triggers:
  ├─ push: code pushed to repository
  ├─ pull_request: PR opened/updated
  ├─ schedule: runs on schedule (cron)
  ├─ workflow_dispatch: manual trigger
  ├─ release: when release created
  ├─ issues: when issue opened
  └─ Other repository events

Example:
  on:
    push:
      branches: [main, develop]
    schedule:
      - cron: '0 2 * * *'  # 2 AM daily

CONCEPT 7: ENVIRONMENT VARIABLES
─────────────────────────────────

What is it?
  └─ Variables accessible during workflow execution

Define in workflow:
  env:
    REGISTRY: docker.io
    IMAGE_NAME: wine-app

Use in jobs:
  echo $REGISTRY
  echo $IMAGE_NAME

CONCEPT 8: SECRETS
──────────────────

What is it?
  └─ Encrypted variables for sensitive data

Use for:
  ├─ API keys
  ├─ Tokens
  ├─ Passwords
  ├─ Database credentials
  └─ Private information

Access in workflow:
  ${{ secrets.DOCKER_USERNAME }}
  ${{ secrets.HEROKU_API_KEY }}

📚 PART 8: SETTING UP SECRETS
================================================================================

GitHub Secrets are encrypted variables for sensitive data.

WHERE TO SET SECRETS:
─────────────────────

On GitHub website:
  1. Go to repository
  2. Settings (tab)
  3. Security → Secrets and variables → Actions
  4. Click "New repository secret"
  5. Add secret name and value

SECRETS YOU NEED FOR YOUR PROJECT:
───────────────────────────────────

1. DOCKER_USERNAME
   ├─ Value: Your Docker Hub username
   ├─ Purpose: Username for docker push

2. DOCKER_PASSWORD
   ├─ Value: Your Docker Hub password or token
   ├─ Purpose: Login to Docker Hub
   ├─ Better: Use access token instead of password
   ├─ Create at: https://hub.docker.com/settings/security

3. HEROKU_API_KEY
   ├─ Value: Your Heroku API key
   ├─ Purpose: Login to Heroku
   ├─ Get from: https://dashboard.heroku.com/account/applications/authorizations/new

4. HEROKU_EMAIL
   ├─ Value: Your Heroku account email
   ├─ Purpose: Email for Heroku login

HOW TO CREATE HEROKU API KEY:
──────────────────────────────

Step 1: Go to https://dashboard.heroku.com/account/applications/authorizations
Step 2: Click "Create authorization"
Step 3: Give it a name (e.g., "GitHub Actions")
Step 4: Copy the token
Step 5: Add to GitHub Secrets as HEROKU_API_KEY

HOW TO CREATE DOCKER TOKEN:
─────────────────────────────

Step 1: Go to https://hub.docker.com/settings/security
Step 2: Click "New Access Token"
Step 3: Give it a name (e.g., "github-actions")
Step 4: Copy the token
Step 5: Add to GitHub Secrets as DOCKER_PASSWORD

📚 PART 9: ACCESSING SECRETS IN WORKFLOWS
================================================================================

In workflow files, access secrets like this:

${{ secrets.SECRET_NAME }}

Examples:

Docker login:
  uses: docker/login-action@v1
  with:
    username: ${{ secrets.DOCKER_USERNAME }}
    password: ${{ secrets.DOCKER_PASSWORD }}

Heroku deployment:
  uses: akhileshns/heroku-deploy@v3.12.12
  with:
    heroku_api_key: ${{ secrets.HEROKU_API_KEY }}
    heroku_email: ${{ secrets.HEROKU_EMAIL }}

IMPORTANT SECURITY NOTES:
─────────────────────────

⚠️ NEVER:
  ├─ Commit secrets to GitHub
  ├─ Put passwords in workflow files
  ├─ Hardcode API keys
  ├─ Share secrets with others
  └─ Use in logs/output

✅ DO:
  ├─ Use GitHub Secrets
  ├─ Access via ${{ secrets.NAME }}
  ├─ Keep backup of keys securely
  ├─ Rotate secrets periodically
  └─ Restrict access to repository

📚 PART 12: COMMON GITHUB ACTIONS ERRORS & SOLUTIONS
================================================================================

ERROR 1: "Secrets not found"
──────────────────────────

Error message:
  Docker username is empty

Cause: Secrets not configured

Solution:
  1. Go to Settings → Secrets and variables
  2. Add DOCKER_USERNAME secret
  3. Add DOCKER_PASSWORD secret
  4. Commit and push again

ERROR 2: "Authentication failed"
────────────────────────────────

Error message:
  Error response from daemon: unauthorized

Cause: Wrong credentials

Solution:
  ├─ Check Docker Hub username/password
  ├─ Use access token not password
  ├─ Verify secret value matches
  ├─ Recreate secret if needed

ERROR 3: "Tests failed"
──────────────────────

Error message:
  pytest: command not found

Cause: Dependencies not installed

Solution:
  └─ Add step: pip install pytest

ERROR 4: "Permission denied"
──────────────────────────

Error message:
  Permission denied: /home/runner/work

Cause: File permissions

Solution:
  ├─ Add chmod in workflow
  ├─ Or use different path
  ├─ Or adjust file permissions

ERROR 5: "Timeout"
──────────────────

Error message:
  The operation timed out

Cause: Job taking too long

Solution:
  ├─ Optimize tests (run faster)
  ├─ Parallel jobs: runs-on: ubuntu-latest
  ├─ Increase timeout limit
  ├─ Remove slow steps

ERROR 6: "Out of storage"
──────────────────────────

Error message:
  No space left on device

Cause: Too many artifacts or temp files

Solution:
  ├─ Clean up temp files in workflow
  ├─ Don't save large artifacts
  ├─ Use matrix strategy efficiently
  ├─ Remove old artifacts

ERROR 7: "Branch protection rules block merge"
──────────────────────────────────────────────

Cause: Workflow didn't pass

Solution:
  ├─ Fix code/tests
  ├─ Make workflow pass
  ├─ Then pull request can merge

ERROR 8: "Docker push fails"
───────────────────────────

Error message:
  Error pushing image

Cause: Docker Hub login failed

Solution:
  ├─ Check secrets are correct
  ├─ Verify Docker Hub credentials
  ├─ Check internet connection
  ├─ Try local docker login

📚 PART 13: ADVANCED GITHUB ACTIONS FEATURES
================================================================================

FEATURE 1: CONDITIONAL STEPS
────────────────────────────

Run step only if condition met:

Example - only deploy on success:
  - name: Deploy
    if: success()
    run: echo "Deploying..."

Common conditions:
  if: success()        (previous step succeeded)
  if: failure()        (previous step failed)
  if: always()         (always run this step)
  if: github.ref == 'refs/heads/main'  (only on main)

FEATURE 2: MATRIX STRATEGY
─────────────────────────

Test on multiple Python versions:

strategy:
  matrix:
    python-version: [3.7, 3.8, 3.9, 3.10, 3.11]

Then use:
  with:
    python-version: ${{ matrix.python-version }}

Result: Job runs on all Python versions in parallel!

FEATURE 3: ARTIFACTS
────────────────────

Save files from workflow:

Upload:
  - uses: actions/upload-artifact@v2
    with:
      name: test-reports
      path: test-results/

Download:
  - uses: actions/download-artifact@v2
    with:
      name: test-reports

Use case:
  ├─ Save test coverage reports
  ├─ Save build outputs
  ├─ Save logs
  ├─ Share between jobs

FEATURE 4: CACHING
──────────────────

Cache dependencies to speed up:

  - uses: actions/cache@v2
    with:
      path: ~/.cache/pip
      key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }}

Benefits:
  ├─ Second run much faster
  ├─ Pip packages cached
  ├─ Avoid downloading repeatedly
  └─ Save time & bandwidth

FEATURE 5: SCHEDULED WORKFLOWS
──────────────────────────────

Run on schedule (like cron):

on:
  schedule:
    - cron: '0 2 * * *'  # 2 AM every day
    - cron: '0 */4 * * *' # every 4 hours

Use cases:
  ├─ Nightly tests
  ├─ Backup data daily
  ├─ Security scans periodically
  ├─ Performance testing
  └─ Maintenance tasks

FEATURE 6: MANUAL TRIGGER
─────────────────────────

Manually trigger workflow from GitHub UI:

on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy to'
        required: true
        default: 'staging'

Then in workflow:
  run: echo "Deploying to ${{ github.event.inputs.environment }}"

Benefit: Deploy manually without pushing code!

FEATURE 7: NOTIFICATIONS
────────────────────────

Send notifications when workflow fails:

Option 1: Slack
  - uses: slackapi/slack-github-action@v1
    with:
      webhook-url: ${{ secrets.SLACK_WEBHOOK }}

Option 2: Email (GitHub native)
  └─ GitHub sends email automatically

Option 3: Custom with curl
  - run: |
      curl -X POST https://your-api/notify \
        -d "status=${{ job.status }}"

📚 PART 14: BEST PRACTICES FOR GITHUB ACTIONS
================================================================================

BEST PRACTICE 1: Keep Workflows Simple
───────────────────────────────────────
❌ Wrong: One giant workflow with 50 steps
✅ Right: Multiple focused workflows

BEST PRACTICE 2: Use Specific Action Versions
───────────────────────────────────────────────
❌ Wrong: uses: actions/checkout@latest
✅ Right: uses: actions/checkout@v2

BEST PRACTICE 3: Meaningful Job/Step Names
────────────────────────────────────────────
❌ Wrong: name: step1
✅ Right: name: Run unit tests with pytest

BEST PRACTICE 4: Fail Fast
──────────────────────────
Stop early if something breaks:
  fail-fast: true
  └─ Don't waste time testing all Python versions if v3.7 fails

BEST PRACTICE 5: Use Caching
─────────────────────────────
Cache dependencies:
  └─ Dramatically speed up workflows

BEST PRACTICE 6: Don't Hardcode Secrets
──────────────────────────────────────────
❌ Wrong: password: "my-secret-password"
✅ Right: password: ${{ secrets.PASSWORD }}

BEST PRACTICE 7: Use Branch Protection Rules
──────────────────────────────────────────
Require workflows to pass before merge:
  1. Settings → Branches
  2. Add branch protection rule
  3. Require status checks to pass
  4. Select your workflows
  5. Save

Result: Bad code never reaches main!

BEST PRACTICE 8: Document Workflows
─────────────────────────────────────
Add comments in workflow files:
  ```yaml
  # This workflow tests code on every push
  # It runs linting, unit tests, and integration tests
  ```

BEST PRACTICE 9: Monitor Workflow Performance
──────────────────────────────────────────────
Check:
  ├─ How long workflows take
  ├─ Which steps are slow
  ├─ Optimize bottlenecks
  └─ Improve efficiency

BEST PRACTICE 10: Version Your Workflows
──────────────────────────────────────────
When making changes:
  ├─ Test on develop branch first
  ├─ Then merge to main
  ├─ Document changes in README
  └─ Keep history of changes
